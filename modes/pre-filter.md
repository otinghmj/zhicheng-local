# 模式：pre-filter — 采集初筛

## 定位与职责

介于**采集层**（job-scraper）和**分析层**（/zhicheng pipeline）之间的轻量过滤器。

- **输入**：采集脚本产出的简易职位列表（无 JD），即 report.json 中的 `dedupJobs[]`
- **输出**：通过初筛的职位写入 `data/pipeline.md`，过滤掉的记录原因
- **不做**：不抓 JD 详情，不生成报告，不接触 cv.md

> 目标是把采集到的 20-100 条简易数据快速缩减到 5-15 条高价值候选，再交给分析层做完整 A-F 评估。

---

## 读取材料（刻意轻量）

只读以下两个文件，不读 cv.md：

1. `config/profile.yml` → 薪资目标、城市偏好、目标原型、排除关键词
2. `modes/_profile.md` → 叙事方向、目标行业、优先公司类型

---

## 第一步：硬过滤（直接淘汰，不计分）

以下任一条件满足即直接过滤，不进入评分：

| 条件 | 来源字段 | 说明 |
|------|---------|------|
| 职位名包含排除词 | `jobName` | 对照 `search_strategy.filters.exclude_keywords`（实习/兼职/驻场/外包/销售） |
| 薪资上限 < 最低底线 | `salaryDesc` | 解析后月薪上限 < `compensation.minimum`（20k/月） |
| 经验要求完全超出范围 | `jobExperience` | 要求 > `exp_max_years + 2` 年（即要求 10 年以上视为超出） |
| 城市为硬排除城市 | `cityName` | 如用户在 profile 中明确排除的城市（当前无，默认不过滤） |

---

## 第二步：三维评分（总分 5 分）

### 维度 1 — 方向契合度（0-2 分）

对照 `target_roles.primary` / `target_roles.secondary` 判断：

| 分值 | 标准 |
|------|------|
| 2 | 职位名直接命中 primary 目标原型（AI应用工程师、工业AI、知识图谱、工业软件PM、LLM工程师、RAG工程师、AI Agent） |
| 1 | 命中 secondary 目标（技术项目经理、研发项目经理、制造数字化、数字化项目经理）；或职位名+技能标签组合暗示匹配（如"项目管理"+"制造"+"AI"标签） |
| 0 | 与任何目标方向无关（SQE、供应商质量、IQC、质检、采购、HR、行政、销售类） |

### 维度 2 — 薪资匹配度（0-2 分）

将 `salaryDesc` 解析为**月薪等效区间**：
- `25-40K·16薪` → 月均 33k-53k
- `20-35K` → 月均 20k-35k（默认12薪）
- `面议` / 未披露 → 视为 1 分（不确定）

| 分值 | 标准 |
|------|------|
| 2 | 薪资区间与目标区间（25k-45k）有明确重叠，且上限 ≥ 30k |
| 1 | 薪资未披露（面议）；或区间仅轻微低于目标（上限在 20-25k） |
| 0 | 薪资上限 < 20k，或明显低于目标且无弹性空间 |

### 维度 3 — 行业/公司背景加成（0-1 分）

| 分值 | 标准 |
|------|------|
| 1 | `brandIndustry` 或公司标签属于：制造业、工业互联网、工业自动化、智能制造、工业软件、数字化、新能源、半导体；**或** 公司标签显示 AI/大模型/科技属性（适用于 RAG/LLM 方向投互联网公司场景） |
| 0 | 其他行业（不扣分，仅不加分） |

---

## 第三步：城市加权（不独立计分，影响排序）

城市不影响是否通过，但影响通过后的排列顺序：

- `preferred_cities`（上海/深圳/北京/广州/杭州/佛山）中的城市 → 排序靠前
- 其他城市 → 正常排序

---

## 过滤阈值

默认 **≥ 3.0 分**通过，写入 `data/pipeline.md`。

可在 `config/profile.yml` 中通过以下字段自定义：
```yaml
pre_filter:
  threshold: 3.0       # 通过分数线（1-5）
  max_pipeline: 15     # 单次最多写入 pipeline 条数（防止爆量）
```

若通过条数超过 `max_pipeline`，按分数从高到低截取。

### 写入 pipeline.md

通过初筛的条目按原格式写入：

```
- [ ] {url} | {brandName} | {jobName} | {salaryDesc} | {cityName} | {jobExperience} | {jobDegree} | {brandIndustry} | {brandScaleName} | 初筛分:{score}/5
```

末尾追加 `初筛分:{score}/5` 字段，供后续分析层参考。

---

## 调用方式

### 方式 A：job-scraper 采集后自动触发（推荐）

在 job-scraper skill 执行完采集后，Claude 自动：

1. 读取返回的 `reportPath` 对应的 report.json
2. 执行本模式的评分逻辑
3. 只将通过条目写入 pipeline.md（不调用各平台的 rpa-to-pipeline.mjs）

> **因此：job-scraper 采集时必须加 `--skip-pipeline` 参数，由本模式统一接管写入。**

### 方式 B：手动对已有 report.json 补跑

```
/zhicheng pre-filter --report output/hammerspoon/boss-rpa/AI应用工程师/report.json
```

## 与分析层的边界

| 维度 | 初筛层（pre-filter）| 分析层（oferta/auto-pipeline）|
|------|--------------------|-----------------------------|
| 输入数据 | 简易列表（无 JD） | 完整 JD |
| 读取档案 | profile.yml + _profile.md | cv.md + profile.yml + _profile.md |
| 评分维度 | 3 维（方向/薪资/行业） | 6 维 A-F |
| 单条耗时 | 秒级（批量处理） | 分钟级（深度分析）|
| 输出 | 通过/过滤 + 简评 | 完整报告 + PDF + 入库 |
| 决策权重 | 粗筛（宁放勿漏） | 精筛（真实投递决策依据）|

> **宁放勿漏原则**：初筛倾向保守，边界案例（分数在 2.5-3.0）建议放行，让分析层做最终裁决。
